# Ethan task breakdown

## When to use

**命中以下任意一条即可使用本 Artifact：**

- 用户要求把需求 / 功能 / 架构调整拆成开发任务
- 用户提到"依赖关系"、"前置条件"、"先后顺序"、"阻塞"等依赖相关表述
- 用户提到"子任务"、"嵌套拆分"、"层级拆分"等复杂拆分需求
- 用户提到"关键路径"、"并行"、"里程碑"、"排期"等项目管理需求
- 需要输出结构化任务列表（表格 + 依赖图）

## Execution

### 阶段一：理解需求

1. **识别目标**：明确用户需求的核心交付物与验收标准。
2. **识别影响边界**：列出涉及的模块、服务、接口，不发明未存在的系统边界。

### 阶段二：任务拆分

3. **拆成 1~3 天粒度任务**：将需求拆解为可独立描述的原子任务，每个任务控制在 1~3 天工作量。
4. **嵌套子任务拆分（按需）**：当某个任务仍复杂时，将其进一步拆为子任务。
   - 子任务 ID 格式：`T1.1`、`T1.2`（父任务 T1 的子任务）。
   - 子任务粒度仍遵循 1~3 天约束；父任务的粒度 = 其子任务粒度之和。
   - 依赖可跨层级：`T2.1` 可依赖 `T1.3`，不必只依赖同层级。

#### 子任务 vs 独立任务：拆分决策指南

遇到复杂任务时，按以下规则决定是拆子任务还是拆成独立任务：

| 条件 | 建议 |
|------|------|
| 多个部分共享同一交付物，且必须一起交付 | 拆为同一父任务的子任务 |
| 各部分可独立交付、独立验收 | 拆为独立任务 |
| 共享代码模块或数据结构，修改同一组文件 | 拆为子任务（减少冲突） |
| 一部分是另一部分的前置，但属于同一逻辑步骤 | 拆为子任务，建立子任务间 hard 依赖 |
| 子任务数量超过 5 个 | 考虑拆为独立任务组，用里程碑组织 |

### 阶段三：依赖分析（核心能力）

5. **分析子任务依赖关系**：
   - 对每个任务/子任务标注其前置任务（`blocked_by`）和后续任务（`blocks`）。
   - **标注依赖类型**：
     - `hard`：硬依赖，前置任务**必须完成**才能开始。
     - `soft`：软依赖（可选），前置任务完成后质量更高，但**不阻塞启动**。
   - 识别可并行的任务组。
   - 检测循环依赖；若发现，立即升级。

#### 常见依赖模式速查

从需求描述中快速识别以下依赖模式：

| 模式 | 需求中的线索 | 依赖结构 | 示例 |
|------|-------------|---------|------|
| **顺序链** | "先做 A，再做 B，然后 C" | A → B → C（全部 hard） | 定义接口 → 实现服务 → 接入调用方 |
| **汇聚点** | "等 X 和 Y 都完成后再做 Z" | X ↘ ↗ Z；Y ↗（Z blocked_by X, Y） | 前端页面 + 后端 API 都完成后联调 |
| **扇出** | "A 完成后，B、C、D 可以同时做" | A → B, A → C, A → D | 数据模型定义完成后，CRUD 各自并行 |
| **软依赖** | "最好先完成 A，但 B 不一定等" | A ─soft→ B | 设计规范完成后开发更好，但不阻塞 |
| **条件依赖** | "如果走方案一则依赖 X，否则依赖 Y" | 分支标注 | 需用户确认方案后再定 |

#### 跨层级依赖规则

- 子任务可以依赖**其他父任务的子任务**：`T2.1 blocked_by T1.3`（合法）
- 子任务**不应**直接依赖其他**父任务本身**：`T2.1 blocked_by T1`（不规范，应细化到 T1 的某个子任务）
- 父任务之间的依赖 = 其末尾子任务到对方起始子任务的依赖传递

### 阶段四：排期与关键路径

6. **划分里程碑**：按交付阶段将任务分组为里程碑（M1, M2, ...），每个里程碑有明确交付物。
7. **关键路径分析**：
   - 根据硬依赖链计算最长路径（关键路径）。
   - 标注关键路径上的任务（`critical: yes`），这些任务延期会直接导致整体延期。

### 阶段五：输出

8. **输出任务列表与依赖关系**：按以下格式输出。

### 输出格式

```markdown
## 里程碑总览

| 里程碑 | 包含任务 | 交付物 |
|--------|---------|--------|
| M1 | T1, T2 | ... |
| M2 | T3, T4 | ... |

## 任务列表

| ID | 任务名称 | 描述 | 粒度(天) | blocked_by | 依赖类型 | blocks | critical |
|----|---------|------|---------|-----------|---------|--------|----------|
| T1 | ... | ... | 2 | — | — | T2, T3 | yes |
| T1.1 | ... | T1 子任务 | 1 | — | — | T1.2 | yes |
| T1.2 | ... | T1 子任务 | 1 | T1.1 | hard | T2 | yes |
| T2 | ... | ... | 1 | T1 | hard | T4 | yes |
| T3 | ... | ... | 1 | T1 | soft | T4 | no |
| T4 | ... | ... | 2 | T2, T3 | hard | — | yes |

## 依赖关系图（文本）

T1.1 → T1.2 → T2 → T4  ← 关键路径（加粗或标注 ★）
T1 → T3 ─→ T4          （软依赖，虚线）

## 关键路径

★ T1.1 → T1.2 → T2 → T4（总粒度：6 天）

## 可并行任务组

- 并行组 1: T1.1（无前置）
- 并行组 2: T1.2（依赖 T1.1）
- 并行组 3: T2, T3（T2 硬依赖 T1.2，T3 软依赖 T1，可并行）
- 并行组 4: T4（依赖 T2 硬 + T3 软）
```

### 阶段六：验证与维护

9. **验证完整性**：
   - 确认所有 `blocked_by` / `blocks` 指向的任务 ID 存在，无悬空引用。
   - 确认无循环依赖。
   - 确认每个里程碑内的任务依赖不跨后续里程碑（跨里程碑的硬依赖需显式标注并提醒用户）。
   - 确认子任务引用规则合规（不直接依赖其他父任务，细化到子任务级别）。
10. **依赖变更影响分析（按需）**：当用户后续提出任务调整时，重新评估：
    - 新增/删除任务对依赖图的影响。
    - 关键路径是否变化。
    - 受影响的并行分组。

#### 依赖复杂度降级策略

当依赖关系过于复杂（如单个任务 blocked_by 超过 3 个、嵌套超 2 层）时，执行降级：

| 复杂度信号 | 降级操作 |
|-----------|---------|
| 单任务 blocked_by > 3 | 合并部分前置任务为一个大任务，减少汇聚点 |
| 嵌套超过 2 层（T1.1.1） | 将第三层提升为独立任务 |
| 软依赖超过总数的 50% | 重新审视是否所有软依赖都有必要标注 |
| 依赖图形成网状（无明确层级） | 按模块/领域重新分组，再组内建链 |

## 依赖分析快速参考

遇到"子任务依赖关系"类请求时，L0 应直接进入**阶段三**工作流，核心动作：

| 步骤 | 动作 | 输出 |
|------|------|------|
| 1 | 为每个任务标注 `blocked_by` 和 `blocks` | 有向依赖边 |
| 2 | 标注依赖类型 `hard` / `soft` | 每条边的类型 |
| 3 | 识别依赖模式（参考"常见依赖模式速查"） | 模式标注 |
| 4 | 检测循环依赖 | 有环则升级 |
| 5 | 计算关键路径（仅沿 `hard` 边） | 关键路径任务序列 |
| 6 | 识别可并行任务组 | 并行分组列表 |
| 7 | 检查复杂度，必要时降级 | 简化后的依赖图 |
| 8 | 输出依赖关系图 + 任务表 | 标准输出格式 |

## Escalate when

- 缺少必要上下文（无法判断模块边界或依赖方向）
- 当前请求需要新的核心协议（未定义的协议不应假设）
- 发现循环依赖，需要用户确认是否拆分或合并任务
- 跨里程碑的硬依赖导致无法按阶段交付，需用户重新确认里程碑划分
- 请求涉及未定义的系统边界
- 子任务嵌套超过 2 层（T1.1.1），建议重新审视任务粒度或升级为独立需求
- 依赖关系过于复杂，降级策略无法简化，需用户参与确认优先级
- 无法确定某条依赖是 hard 还是 soft，需用户明确业务语义
