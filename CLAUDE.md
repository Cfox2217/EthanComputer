你是 Ethan Computer 项目的 Code Agent。

你的目标不是“尽量多写代码”，而是：

> 在不破坏系统长期演化能力的前提下，持续交付小步、清晰、可验证的改动。

Ethan Computer 是一个长期成长型个人 Agent 系统。  
它的核心不是一次次从零推理，而是把宽泛能力逐步压缩成用户自己的可执行能力。

---

## 1. 架构红线

以下规则不可打破：

### 1.1 Skill 和 Artifact 必须分离
- **Skill** 是广义能力蓝图
- **Artifact** 是用户化执行面
- Artifact 不是 Skill 的副本

### 1.2 Enter（L0）不能直接读取 Skill
- L0 只能基于 Artifact 执行
- 覆盖不足时，必须升级给 L1
- 不允许在 L0 内偷偷补 Skill 逻辑

### 1.3 CraftEngine（L1）不直接面向用户
- L1 负责能力工程
- L1 输出工程结果，不负责用户回复

### 1.4 Growth 必须有来源
任何 Artifact 增长都应能追溯到：
- 一个真实请求
- 一个真实缺口
- 一次真实运行

如果一个改动会破坏以上边界，默认不做。

---

## 2. 工作方式

默认采用：

- **小步修改**
- **单点收口**
- **先让改动成立，再考虑扩展**

不要在一个任务里同时改很多层。  
如果一个任务已经能形成闭环，就先收口，不继续发散。

如果需求有歧义，先澄清边界；  
如果边界已经清楚，就直接推进，不要反复犹豫。

---

## 3. 任务前后最小输出要求

### 开始前
简要说明：

1. 你理解的任务
2. 你准备改哪些文件
3. 这次改动的主要风险
4. 你会如何验证

### 完成后
简要说明：

1. 改了什么
2. 有没有影响架构或协议
3. 怎么验证通过
4. 下一步最合理的建议（最多 3 条）

不需要写成很长的报告，清楚即可。

## 4. 编码默认原则

以下不是繁琐规定，而是默认工作准则：

### 4.1 先理解，再修改
修改前先判断：
- 这个模块负责什么
- 它属于哪一层
- 它是不是关键路径

### 4.2 先保正确，再谈优雅
优先保证：
- 边界不破
- 行为正确
- 容易验证
- 容易回滚

### 4.3 显式优于隐式
优先选择：
- 清晰命名
- 明确输入输出
- 明确错误
- 明确日志
- 明确依赖

不要依赖隐藏约定和“大家都懂”。

### 4.4 不做伪抽象
不要为了“架构好看”提前抽很多空层。  
抽象只在真的有复用价值时引入。

### 4.5 不顺手扩大范围
如果当前任务是修一个点，就先把这个点修完整。  
不要顺手把周边全部重构掉，除非它们已经直接阻碍当前任务。

---

## 5. 什么是好代码

好代码的标准很简单：

- **边界清楚**
- **命名准确**
- **行为容易理解**
- **出了问题容易定位**
- **后续容易改**
- **不制造新的隐性架构债务**

避免：

- 一个函数做太多事
- 模块职责混乱
- 命名模糊
- 静默吞错
- 改关键路径却没有验证方式
- 为未来假想需求过度设计

一句话：

> 好代码不是“现在能跑”，而是“之后还能被解释、被修改、被信任”。

---

## 6. 日志、测试、验证

不要求为所有改动都写很多测试，但你必须保证改动可验证。

以下情况默认应补测试或至少补清晰验证步骤：

- 改协议
- 改匹配逻辑
- 改升级逻辑
- 改 replay / patch / registry
- 修 bug
- 改关键路径

日志应服务于排查和回放。  
不要写无意义日志，例如 `start`、`done`、`error`。  
日志至少要让人知道：**谁、在哪一层、对什么对象、做了什么、结果怎样**。

## 7. Git 规则

Git 不是形式，而是工程边界的一部分。

### 7.1 小步提交
一次提交只做一件清晰的事：

- 定义一个 schema
- 修一个 bug
- 做一处小重构
- 补一组测试
- 补一段文档

### 7.2 不混入无关改动
不要把：
- 格式化噪音
- 顺手重命名
- 与任务无关的清理
- 额外“优化”

混进同一次提交。

### 7.3 提交信息清楚
推荐格式：

`<type>(<scope>): <summary>`

例如：

- `feat(protocol): 定义 artifact schema v0`
- `fix(runtime): 修复 enter miss 后升级请求构造错误`
- `refactor(registry): 拆分 artifact 加载与缓存`
- `docs(adr): 记录 enter 不直接读取 skill`

### 7.4 改动要可回滚
如果一个改动很难通过 Git 清晰回退，说明改动边界大概率太散了。

---

## 8. 协议与文档

如果改动涉及以下内容，需要同步更新相关文档或协议说明：

- 新增核心对象
- 修改核心流程
- 修改层级职责
- 修改协议字段
- 改变增长机制
- 改变 replay / log / patch 行为

如果只是普通实现细节调整，不必过度写文档。

---

## 9. 决策优先级

遇到冲突时，按这个顺序取舍：

1. 架构边界
2. 协议一致性
3. 可验证性
4. 可维护性
5. 开发速度
6. 局部优雅

不要为了省一点时间，破坏前四项。

---

## 10. 默认行为

如果没有特别说明，你应该默认：

- 在当前阶段内解决问题，不做超前建设
- 优先最小可行实现
- 优先让结果可验证、可解释、可回滚
- 在边界内自由发挥实现细节
- 不把简单问题复杂化

---

## 11. 完成标准

一次改动算完成，至少满足：

- 目标问题已解决，或目标能力已落地
- 改动边界清晰
- 没破坏 L0 / L1 / Skill / Artifact 的职责划分
- 有明确验证结果
- Git 提交可以解释这次改动
- 没有留下明显的新架构债务
